# PRD: Obsidian HTML Viewer Plugin (`obsidian-html-viewer`)

## Problem Statement

Obsidian users and developers using automated learning or documentation generation tools (such as the `/teach` skill) often produce rich HTML reference materials, interactive lessons, cheat sheets, and visual documents directly within their vaults. However, existing Obsidian HTML preview plugins sandbox HTML rendering in a manner that prevents loading local relative CSS stylesheets (`<link rel="stylesheet" href="...">`), local images, and local scripts. This causes rendered documents to display unstyled, with broken image links and non-functional interactivity. Furthermore, hyperlinked navigation within these HTML files pointing to Markdown notes (`.md`) or other vault documents fails to integrate with Obsidian's native tab and workspace navigation.

## Solution

Build `obsidian-html-viewer`, a native Obsidian plugin that registers a dedicated custom View type for `.html` files in Obsidian. When an HTML file is opened, the plugin renders the document inside a sandboxed `<iframe>` while dynamically resolving all relative resource paths (CSS stylesheets, images, embedded media, and JavaScript files) to valid Obsidian vault local resource URIs (`vault.getResourcePath`). 

The plugin intercepts link clicks inside the rendered HTML to open vault `.md` notes and `.html` documents natively within Obsidian workspace tabs, while enforcing a strict Content Security Policy (CSP) to restrict external network access. It also provides automatic live reloading when source HTML or linked CSS/JS files change, and supports sandboxed JavaScript execution for interactive documents.

## User Stories

1. As an Obsidian user, I want clicking an `.html` file in the file explorer to open it as a main workspace tab, so that I can view HTML files just like standard notes.
2. As a user viewing an HTML lesson generated by the `/teach` skill, I want local CSS stylesheets in `./assets/style.css` to render properly, so that the lesson has clean typography and layout.
3. As a user viewing an HTML cheat sheet, I want embedded local image assets to display correctly, so that visual diagrams and figures are visible.
4. As a learner navigating an interactive lesson, I want links to `.md` learning records (e.g., `<a href="../learning-records/0001-lesson.md">`) to open as native Obsidian note tabs, so that I can seamlessly transition between HTML reference material and Markdown notes.
5. As a user reading an HTML document with links to other `.html` reference files, I want clicking those links to open the target HTML file in Obsidian, so that I can browse interconnected HTML documentation effortlessly.
6. As a user opening external web links embedded within an HTML document, I want them to open in my system's default web browser, so that external browsing does not disrupt my Obsidian workspace.
7. As a user editing an HTML file or modifying its accompanying CSS stylesheet, I want the rendered HTML view to automatically refresh when files are saved, so that I can see live visual updates immediately.
8. As a user, I want to toggle between the raw HTML source code editor and the rendered HTML preview in the same tab, so that I can inspect and edit HTML markup when necessary.
9. As a security-conscious user, I want the HTML preview to be strictly sandboxed with a local-only Content Security Policy, so that untrusted HTML files cannot make unauthorized external network requests or access sensitive APIs.
10. As a developer using interactive HTML widgets, I want sandboxed JavaScript execution to be enabled within the iframe, so that interactive diagrams, toggles, and client-side scripts function properly.
11. As a user working on mobile (iOS/Android) or desktop (macOS/Windows/Linux), I want local file paths to resolve correctly across all operating systems, so that my HTML notes are accessible on any device.
12. As a user closing and reopening Obsidian, I want my active HTML viewer tabs and view states to be remembered, so that my workspace layout is preserved across sessions.

## Implementation Decisions

### Architectural Components
- **`HtmlView` (Custom ItemView)**: Extends Obsidian's native `ItemView` with view type `html-viewer-view`. Registered in the plugin lifecycle (`registerView`) to handle files with `.html` extension.
- **View State Management**: Supports two view modes within the leaf: `preview` (sandboxed iframe container) and `source` (Obsidian text editor), with a ribbon header icon to switch between modes.
- **Path Resolution & DOM Transformation**: Prior to populating the iframe, the raw HTML string is parsed using `DOMParser`. All relative attributes (`href`, `src`, `poster`, `data`) on elements (`link`, `script`, `img`, `video`, `audio`, `iframe`, `a`) are resolved relative to the active HTML file's vault folder path and converted into valid local resource URIs using Obsidian's `vault.getResourcePath(tfile)`.
- **Security & Content Security Policy (CSP)**: The iframe is instantiated with `sandbox="allow-scripts allow-forms allow-same-origin"`. A strict `<meta http-equiv="Content-Security-Policy">` tag is dynamically injected into the `<head>` of the processed HTML, restricting resource loading (`default-src`, `script-src`, `style-src`, `img-src`) strictly to `app://local/` and Obsidian custom local protocol schemes.
- **IPC Link Interception**: A tiny inline script is injected into the rendered HTML inside the iframe. It attaches a global `click` event listener to `document.body`. When an `<a>` element is clicked:
  - Relative or absolute vault file paths (`.md`, `.html`, `.pdf`, etc.) are intercepted, `preventDefault()` is called, and a message is posted to the parent frame via `window.parent.postMessage({ type: 'obsidian-navigate', linkText: ... }, '*')`.
  - External links (`http://`, `https://`, `mailto:`) are passed to Obsidian's `Electron.shell.openExternal` or `window.open`.
- **Parent Event Listener**: The `HtmlView` listens for `message` events from the window. Upon receiving an `obsidian-navigate` payload, it uses Obsidian's `workspace.openLinkText` to open the referenced file in the workspace.
- **Vault Modification Watcher**: Registers `this.registerEvent(this.app.vault.on('modify', ...))`. When a modified `TFile` matches the currently viewed HTML file or any of its detected local CSS/JS asset dependencies, the `HtmlView` re-renders its iframe content smoothly.

## Testing Decisions

### Seam Strategy
Testing will be anchored at high-level logical seams to ensure reliable behavior without over-coupling to internal Obsidian implementation details:
1. **HTML Path Processor Seam (`resolveHtmlAssets`)**: Pure function taking `(rawHtml: string, currentFilePath: string, getResourcePathFn: (path: string) => string)`. This isolates all DOM rewriting, relative path calculation, and CSP injection for unit testing.
2. **IPC Navigation Message Parser Seam (`handleIpcNavigation`)**: Function taking postMessage payload events and mapping them to expected vault navigation targets.
3. **View State Lifecycle Seam**: Mocking `ItemView` lifecycle methods to verify view mode toggling (`source` vs `preview`) and file modification listeners.

### Test Coverage
- Unit tests for relative path resolving (testing `./style.css`, `../assets/logo.png`, `/root-asset.js`, and anchor fragment IDs like `#heading-1`).
- Verification that CSP meta tag headers are correctly formatted and injected.
- Behavioral test verifying external URLs bypass internal workspace navigation and call external window opening handlers.

## Out of Scope

- Fetching or executing remote CDN scripts and fonts (all assets must be locally present in the vault or bundled).
- Direct WYSIWYG visual editing of HTML content inside the preview iframe.
- Automatic conversion of HTML files to Markdown or vice versa.
- PDF rendering inside the HTML iframe (PDFs linked in HTML will open in Obsidian's native PDF viewer).

## Further Notes

- Designed specifically to seamlessly support HTML lesson folders, glossaries, cheat sheets, and learning records generated by skills such as `/teach` (`./lessons/*.html`, `./reference/*.html`, `./assets/*`, `./learning-records/*.md`).
